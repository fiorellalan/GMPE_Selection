gmpe.py

GMPE computation engine — OpenQuake GSIM wrapper with automatic frequency selection. Standalone module used by compute_targetspectra.py. The GMPE list it evaluates is prepared by the GMPE Selection GUI.

⚙️ Overview

gmpe.py provides the gmmtools class, which wraps OpenQuake's hazardlib.gsim library. It computes ground‑motion prediction equation (GMPE) spectra given a scenario (magnitude, distance, Vs30, etc.) and returns mean and ±1σ spectral accelerations on a frequency grid.

Core methods:

  • computegmpe() — compute a single GMPE spectrum
  • compute_batch() — compute multiple GMPEs in batch, auto‑computing missing geometry (Rjb, Rrup, width, ztor)
  • read_gmpeinput() — build OpenQuake context objects from scenario parameters

🔬 Algorithm

Step 1 — Parameter Storage

computegmpe() receives scenario parameters and stores them as instance attributes (mag, depth, Epi, vs30, dip, rake, Rjb, Rrup, Rx, etc.).

Step 2 — Automatic Frequency Selection

The GMPE's native spectral periods are read from its COEFFS table via _get_gmpe_native_periods(). Only periods overlapping the user's requested frequency range are kept — no extrapolation, no NaNs.

Step 3 — Context Building

read_gmpeinput() creates three OpenQuake context objects: SitesContext, RuptureContext, and DistancesContext, with automatic inference of z1pt0 and z2pt5 from Vs30.

Step 4 — Spectrum Computation

For each SA period, gmpe.get_mean_and_stddevs() is called. Results are exponentiated (natural‑log space → linear).

Step 5 — Frequency Reversal Guard

Since native periods are sorted increasingly, 1/periods gives decreasing frequencies. The code detects this and reverses arrays if needed so that output frequencies are monotonically increasing.

Step 6 — Range Warning

If the user's requested frequency range extends beyond the GMPE's native range, a warning is printed and the result is truncated to the native range.

native_periods = self._get_gmpe_native_periods(gmpe_inst)
if native_periods is None:
    compute_periods = 1.0 / freq       # fallback
else:
    mask = (native_periods >= p_min_user) & (native_periods <= p_max_user)
    compute_periods = native_periods[mask]
    if len(compute_periods) == 0:
        raise ValueError("GMPE '...' native periods don't overlap with user range")
⚠ Why this matters: Each GMPE defines its own valid period range (e.g. 0.01–10 s). Computing SA outside this range would require extrapolation and produce unreliable results. The code never extrapolates — it restricts computation to the intersection of user demand and GMPE capability.

Example from real runs:

ScenarioUser range (Hz)Common GMPE range (Hz)Points
HF_SMS0.2500 – 33.00000.2512 – 20.000088 / 100

The common frequency range is the intersection of all selected GMPEs' native ranges — only frequencies supported by every GMPE are reported. This guarantees no interpolated or extrapolated values in the mean / median / percentile curves.

📐 Automatic z1pt0 & z2pt5 Inference

When z1pt0 or z2pt5 are not explicitly provided, they are automatically inferred from Vs30 using the Boore et al. (2011) relations:

Vs30 range (m/s)z1pt0 formulaz2pt5 formula
Vs30 < 180exp(6.745) × 0.001 km(519 + 3.595 × z1pt0_m) × 0.001 km
or (519 + 3.595 × Vs30) × 0.001 km if z1pt0 unavailable
180 ≤ Vs30 < 500exp(6.745 − 1.35 × ln(Vs30/180)) × 0.001 km
Vs30 ≥ 500exp(5.394 − 4.48 × ln(Vs30/500)) × 0.001 km
💡 Note: z1pt0 and z2pt5 are the depths (in km) to the 1.0 km/s and 2.5 km/s shear‑wave velocity horizons. They are required by several GMPEs (e.g. Chiou & Youngs 2014, Campbell & Bozorgnia 2014) and significantly influence the predicted ground motion at sediment sites.

📦 Batch Computation — compute_batch()

The compute_batch() method extends computegmpe() to multiple GMPEs with automatic geometry inference:

  • Auto‑computation of Rjb: defaults to epi if not provided
  • Auto‑computation of width: from magnitude and rake angle using empirical scaling relations
  • Auto‑computation of ztor: from depth, width and dip
  • Auto‑computation of Rrup: from ztor and Rjb
  • Individual GMPE failures are caught and reported in an errors array — one failing GMPE does not block the rest
# Example usage
tool = gmpe.gmmtools()
result = tool.compute_batch(
    gmpe_list=["AbrahamsonEtAl2014", "BooreEtAl2014"],
    freq=freq_array, mag=5.6, depth=7.0, epi=0.0, vs30=980.0,
    dip=90.0, rake=0.0
)
# result = {"results": {"AbrahamsonEtAl2014": [mean, sig1m, sig1p, nat_freq], ...},
#           "errors": []}

Returned dict structure per GMPE:

KeyTypeDescription

sig1plistMean + 1σ SA (g)

ColumnDescription

DefinedForIMTsSpace‑separated list of intensity measure types
DefinedForStdDevsSpace‑separated list of standard deviation types

The catalogue is auto‑generated by scanning all available GSIMs via gsim.get_available_gsims(). If the file does not exist, it is created automatically by launching the OpenQuake Python environment in a subprocess.

🚀 Quick Reference

# 1. Import
import gmpe
tool = gmpe.gmmtools()

# 2. Single GMPE
mean, sig1m, sig1p, dists, zone, nat_freq = tool.computegmpe(
    "BooreEtAl2014", freq, mag=5.6, depth=7.0, Epi=0.0, vs30=980.0,
    dip=90.0, rake=0.0
)

# 3. Batch (auto‑geometry)
result = tool.compute_batch(
    gmpe_list=["BooreEtAl2014", "ChiouYoungs2014"],
    freq=freq_array, mag=5.6, depth=7.0, epi=0.0, vs30=980.0
)
mean_array = result["results"]["BooreEtAl2014"][0]  # [mean, sig1m, sig1p, nat_freq]
